# Navalia v1.2 — Agent Capabilities & Owner Operations (Design Spec)

**Status:** approved design — feeds the v1.2 implementation plan.
**Date:** 2026-07-03

## Overview

Three features, one coherent epic. They share a small owner-configuration
surface (registered phone numbers + notification settings), so they ship as one
spec and one plan with three independent task groups.

1. **Voice-message understanding (STT)** — the WhatsApp bot transcribes voice
   notes and answers them like text.
2. **Owner new-appointment notifications** — a dashboard notification center
   (source of truth) plus an opt-in WhatsApp push.
3. **Owner admin channel over WhatsApp** — the owner runs internal (copilot)
   operations by messaging the bot from a pre-registered number; sensitive
   actions require a short-lived PIN shown in the web dashboard.

**Build order:** A (shared foundation) → C (notifications) → D (admin channel)
→ B (STT). B is independent and can move anytime.

## Global Constraints

Every task inherits these (from `CLAUDE.md` / existing code):

- **Tenant scoping:** every query takes an explicit `barbershopId`. No "current
  tenant" global. New repository functions must include `where: { barbershopId }`.
- **Result pattern:** domain use cases return
  `{ ok: true, data: T } | { ok: false, error: string }`; never throw for
  control flow. Server actions return the existing
  `ActionResult<T> = { ok: true; data?: T } | { ok: false; error: string }`
  (`src/modules/tenancy/settings-actions.ts:11`).
- **AI tool rules:** validate tenant/identity from server context, never from
  model output; sensitive tools return `pendingAction` instead of executing;
  log every call to `AiActionLog`.
- **Stack:** Next.js 16 (App Router, `proxy.ts`, standalone), Prisma 7
  (`prisma.config.ts`, `npx prisma migrate dev`), Better Auth, Tailwind v4 +
  shadcn on Base UI. UI copy pt-BR; code/docs English.
- **Design tokens:** no hardcoded colors — use the CSS custom properties in
  `src/app/globals.css`.
- **Testing:** Vitest. Unit tests need no `DATABASE_URL`; integration tests use
  `describe.skipIf(!process.env.DATABASE_URL)`.
- **Phone normalization:** reuse `normalizePhone` from
  `src/modules/whatsapp/evolution-client.ts` (strips `@s.whatsapp.net`,
  prefixes `55` for 10–11 digit numbers) everywhere phones are compared/stored.

## A. Shared foundation — data model & owner config

### A.1 Prisma schema changes (`prisma/schema.prisma`)

**New model `Notification`:**

```prisma
model Notification {
  id           String   @id @default(cuid())
  barbershopId String
  type         String   // APPOINTMENT_CREATED (extensible later)
  appointmentId String?
  readAt       DateTime?
  createdAt    DateTime @default(now())

  barbershop  Barbershop  @relation(fields: [barbershopId], references: [id])

  @@index([barbershopId, readAt])
  @@index([barbershopId, createdAt])
}
```

**New fields on `Barbershop`** (model at `schema.prisma:51-76`):

```prisma
  // Owner admin channel (feature D)
  adminPhones        String[]  @default([])
  adminPinHash       String?
  adminPinExpiresAt  DateTime?
  // Owner notification push (feature C)
  ownerNotifyPhone   String?
  notifyOwnerWhatsapp Boolean  @default(false)
  // relation
  notifications      Notification[]
```

**New fields on `WhatsappConversation`** (model at `schema.prisma:249-259`),
used only by the admin flow:

```prisma
  pendingActionId        String?
  pendingActionExpiresAt DateTime?
```

Conventions to match existing schema: `cuid()` ids, `createdAt @default(now())`,
`updatedAt @updatedAt`, explicit `@@index`. `String[]` maps to Postgres
`text[]` (supported by the PrismaPg adapter).

**Migration:** `npx prisma migrate dev --name v1_2_notifications_admin_channel`.
All new columns are nullable or defaulted, so the migration is additive and safe
on existing rows. No enum changes (`AiActionLog.channel` and `Notification.type`
are plain `String`).

### A.2 Channel type (`src/modules/ai/types.ts:5`)

```typescript
export type Channel = 'WHATSAPP' | 'AI_WEB' | 'COPILOT' | 'WHATSAPP_ADMIN'
```

`AiActionLog.channel` is already `String`, so no DB change is needed to log the
new channel value.

### A.3 Owner-config server actions (`src/modules/tenancy/settings-actions.ts`)

Follow the `saveShopSettings` pattern (`settings-actions.ts:29-67`): Zod schema,
`requireOwner()` guard, empty-patch guard, `prisma.barbershop.update`, return
`ActionResult`.

- `saveAdminChannelConfig({ adminPhones: string[], ownerNotifyPhone?: string, notifyOwnerWhatsapp: boolean })`
  - Normalize every phone with `normalizePhone`.
  - **Reject** any admin/notify phone that equals `normalizePhone(barbershop.phone)`
    when `barbershop.phone` is set — the owner must message the bot from a
    *different* personal number, not the barbershop's own WhatsApp line. (The
    Evolution instance's connected number is not stored directly; `barbershop.phone`
    is the practical anchor.)
  - Dedupe `adminPhones`.
- `generateAdminPin()`
  - Guard `requireOwner()`.
  - Generate a 6-digit PIN with `crypto.randomInt(0, 1_000_000)` zero-padded.
  - Store `adminPinHash = sha256(pin)` and `adminPinExpiresAt = now + 5 min`.
  - Return the **plaintext** PIN in the action result for one-time display.
    The plaintext is never persisted.

### A.4 Settings UI

Add an **"WhatsApp Admin & Notificações"** card to
`src/app/(dashboard)/dashboard/configuracoes/page.tsx` (server component using
`requireOwner()`), rendered by a client component:

- Manage `adminPhones` (add/remove chips), `ownerNotifyPhone`, and a
  `notifyOwnerWhatsapp` toggle → calls `saveAdminChannelConfig`.
- A **"Gerar PIN"** button → calls `generateAdminPin`, shows the 6-digit code
  with a 5-minute countdown and copy affordance. Never renders a stored PIN
  (there is none) — only the freshly returned plaintext.

## B. Voice-message understanding (STT)

### B.1 Evolution media download (`src/modules/whatsapp/evolution-client.ts`)

Add a method to the exported `evolution` object (existing methods at
`evolution-client.ts:132-260`; helper `apiFetch<T>` at `:90-126`; base URL
`process.env.EVOLUTION_URL` default `http://127.0.0.1:8080`; `apikey` header):

```typescript
getBase64FromMedia(
  instanceName: string,
  rawMessage: Record<string, unknown>,
): Promise<Result<{ base64: string; mimetype: string }>>
```

Implementation: `POST /chat/getBase64FromMediaMessage/{instanceName}` with body
`{ message: rawMessage, convertToMp4: false }`. Evolution decrypts the media
server-side and returns base64 — this avoids handling WhatsApp's encrypted
media URL + `mediaKey` ourselves.

> **Probe first (plan Task B.0):** the exact request/response shape of
> `getBase64FromMediaMessage` must be confirmed against the running v2.3.7
> container by sending a real voice note and inspecting the response. Adjust the
> body key (`message` vs `key`) and response field (`base64`) to match. This is
> a verify-external-API step, not a placeholder.

### B.2 Parser returns message kind + raw (`src/modules/whatsapp/pipeline.ts`)

`parseMessagesUpsert` (`pipeline.ts:66-130`) currently returns
`{ instanceName, fromPhone, text, messageId } | null` and reads text only from
`message.conversation` / `message.extendedTextMessage.text`. Extend the return
type to carry enough for media handling:

```typescript
{
  instanceName: string
  fromPhone: string
  text: string | null
  messageId: string
  kind: 'text' | 'audio' | 'other'
  rawMessage: Record<string, unknown> | null   // the inbound `message` object
  audioSeconds: number | null                   // from audioMessage.seconds
}
```

- `kind = 'audio'` when `message.audioMessage` is present.
- `text` stays `null` for audio (transcription fills it later).
- All existing null-return guards (fromMe, groups, status broadcast, malformed)
  are unchanged.

The webhook route (`src/app/api/webhooks/evolution/route.ts:107-143`) already
has the raw message object in scope and passes it into `parseMessagesUpsert`; no
route change beyond forwarding the richer parse result to `handleInboundMessage`.

### B.3 Transcription module (`src/modules/whatsapp/transcribe.ts`, new)

```typescript
export async function transcribeAudio(args: {
  base64: string
  mimetype: string
  contextPrompt?: string   // service + professional names for proper-noun accuracy
}): Promise<Result<{ text: string }>>
```

- Uses the existing OpenAI client (`src/lib/openai.ts`,
  `getOpenAIClient` / `isOpenAIConfigured`).
- Model from `process.env.OPENAI_TRANSCRIBE_MODEL`, default
  **`gpt-4o-mini-transcribe`** (cheaper + better pt-BR/noisy accuracy; the
  `contextPrompt` seeds proper nouns). Swappable to `whisper-1` via env.
- Convert base64 → file with the SDK `toFile(Buffer.from(base64, 'base64'), 'audio.ogg', { type: mimetype })`.
- Call `client.audio.transcriptions.create({ model, file, language: 'pt', prompt: contextPrompt })`.
- Returns `{ ok: false }` on any error; caller falls back to `NON_TEXT_REPLY`.

### B.4 Pipeline audio branch (`handleInboundMessage`, `pipeline.ts:222-377`)

Replace the current non-text branch (`pipeline.ts:341-346`, which sends
`NON_TEXT_REPLY` for everything non-text) with:

1. If `kind === 'audio'`:
   - If `audioSeconds != null && audioSeconds > MAX_AUDIO_SECONDS` (120) → send
     `"Áudio muito longo — mande até 2 minutos ou escreva."`, persist, return.
   - `getBase64FromMedia` → on error, send `NON_TEXT_REPLY`, persist, return.
   - Build `contextPrompt` from the shop's active services + professional names.
   - `transcribeAudio` → on error, send `NON_TEXT_REPLY`, persist, return.
   - Log the transcription to `AiActionLog` (`toolName: 'transcribeAudio'`,
     `channel: 'WHATSAPP'`, `status: EXECUTED | ERROR`) for cost/observability.
   - Set `text = transcript` and **fall through to the existing text path**
     (persist as inbound, rate limit, debounce, `flushToAI`). Persist the inbound
     message content as the transcript so history + audit read naturally.
2. If `kind === 'other'` (image/sticker/etc.) → current `NON_TEXT_REPLY`
   behavior, unchanged.

Audio counts against the existing WhatsApp rate limit
(`rl:wa:${shop.id}:${fromPhone}`, 30/300) since it flows through the same path.

`MAX_AUDIO_SECONDS` is a module constant next to `DEBOUNCE_MS` (`pipeline.ts:17`).

## C. Owner new-appointment notifications

### C.1 Emit on creation (`src/modules/booking/create-appointment.ts:163-346`)

`createAppointment` is the single chokepoint — all sources converge here:
`createPublicAppointment` (`public-actions.ts:172`, `PUBLIC_PAGE`),
`createAppointmentAdmin` (`admin-actions.ts:446`, `ADMIN`), and the public-tools
`createAppointment` tool (`public-tools.ts:318`, `WHATSAPP` / `AI_WEB`).

Inside the existing Serializable transaction, right after the appointment +
`AuditLog` are created (`create-appointment.ts:290-321`), create the
notification row:

```typescript
await tx.notification.create({
  data: { barbershopId: args.tenantId, type: 'APPOINTMENT_CREATED', appointmentId: appointment.id },
})
```

This keeps the write atomic with the booking and covers every source at once.

### C.2 WhatsApp push (best-effort, post-commit)

After the transaction commits and the `Result` is assembled, but before
returning, run a **best-effort** push that must never affect the booking result:

- New module `src/modules/notifications/push.ts`:
  `pushNewAppointmentToOwner(tenantId, appointmentId): Promise<void>` — loads the
  barbershop; if `notifyOwnerWhatsapp && ownerNotifyPhone && whatsappStatus === 'CONNECTED'`,
  formats a short pt-BR summary (customer, service, professional, date, time)
  and calls `evolution.sendText(evolutionInstanceId, ownerNotifyPhone, msg)`.
  Swallows all errors.
- `create-appointment.ts` calls it wrapped in `try/catch` and awaited (so it
  completes in the serverless invocation) but its failure is ignored. To keep
  the booking engine's core pure and unit-testable, the push call sits **outside**
  the transaction and behind the config gate — with no config it is a no-op, so
  existing engine tests are unaffected.

> Dependency direction: `booking → notifications → whatsapp`. No cycle.

### C.3 Dashboard notification center

- **Queries** (`src/modules/notifications/queries.ts`):
  `getUnreadCount(tenantId): Promise<number>`,
  `listRecent(tenantId, limit = 20): Promise<NotificationView[]>` (joins
  `appointment → customer/service/professional` for display),
  `markRead(tenantId, ids: string[]): Promise<void>` (tenant-fenced `updateMany`),
  `markAllRead(tenantId): Promise<void>`.
- **Polling API** (`src/app/api/notifications/route.ts`):
  `GET` → `{ unread: number, items: NotificationView[] }`, auth via
  `auth.api.getSession` + user→barbershop resolve (same pattern as
  `src/app/api/ai/copilot/route.ts:38-56`). `POST` (or `PATCH`) → mark read.
- **UI** (`NotificationBell` client component in the dashboard header/layout):
  polls `GET /api/notifications` every ~30s, shows an unread badge, a dropdown
  list (customer + service + time, relative "há N min"), and mark-read /
  mark-all-read. Polling only — no SSE in v1.2 (the query/API boundary lets us
  add SSE later without touching the UI contract).

## D. Owner admin channel over WhatsApp

### D.1 Shared confirm-action module (refactor)

Extract the atomic-claim + execute + log + audit logic currently inline in
`src/app/api/ai/copilot/confirm/route.ts:67-206` into:

```typescript
// src/modules/ai/confirm-action.ts
export async function confirmSensitiveAction(args: {
  actionId: string
  barbershop: { id: string; timezone: string }
  userId: string
  channel: Channel          // 'COPILOT' (web route) | 'WHATSAPP_ADMIN' (bot)
  reject?: boolean
}): Promise<AiResult<{ toolName: string; output: unknown }>>
```

It performs the same steps the route does today:
- Tenant-fenced `findFirst` of the `AiActionLog`.
- Atomic claim: `updateMany({ where: { id, barbershopId, status: 'PENDING_CONFIRMATION' }, data: { status: reject ? 'REJECTED' : 'PROCESSING' } })`; if `count !== 1` → already processed (409-equivalent error).
- On confirm: rebuild the tool via `buildCopilotTools({ id, timezone }, 'OWNER')`,
  find `toolDef` by `toolName && sensitive === true`, execute with the **stored**
  `input` and a server-built `ctx` (`{ tenantId, channel, userId }`), then set
  `status: 'CONFIRMED'`, `confirmedAt`, and `output`; write `AuditLog`
  (`COPILOT_ACTION_CONFIRMED` / `COPILOT_ACTION_REJECTED`).

The existing web route becomes a thin wrapper: session → `requireOwner`-style
OWNER check → `confirmSensitiveAction({ ..., channel: 'COPILOT' })`. Behavior and
tests for the web path are preserved. The channel passed through is used only for
the `ctx` and logging; the admin path passes `WHATSAPP_ADMIN`.

### D.2 Admin routing in the pipeline (`handleInboundMessage`)

After the barbershop is resolved and the billing gate passes
(`pipeline.ts:234-329`), branch:

- If `normalizePhone(fromPhone)` is in `barbershop.adminPhones` → **admin flow**
  (below).
- Else → the existing customer flow, unchanged.

Resolve the OWNER user once for the admin flow:
`prisma.user.findFirst({ where: { barbershopId: shop.id, role: 'OWNER' } })` →
its `id` is used for `ctx.userId` and audit. If no OWNER user exists, fall back
to the customer flow (defensive).

Admin flow uses its own conversation row (the admin phone is a distinct
`customerPhone` value under the existing `@@unique([barbershopId, customerPhone])`),
its own debounce key `wa:admin:${shop.id}:${fromPhone}`, and its own rate limit
`rl:wa:admin:${shop.id}:${fromPhone}` (30/300).

### D.3 Admin assistant turn

When flushing an admin turn to the AI:

```typescript
runAssistant({
  channel: 'WHATSAPP_ADMIN',
  tenantId: shop.id,
  history,
  userMessage,
  tools: buildCopilotTools({ id: shop.id, timezone: shop.timezone }, 'OWNER'),
  systemPrompt: adminWhatsAppSystemPrompt({ name: shop.name }, today),
  ctx: { tenantId: shop.id, channel: 'WHATSAPP_ADMIN', userId: ownerUserId },
})
```

- New prompt builder `adminWhatsAppSystemPrompt(shop, today)` in
  `src/modules/ai/prompts.ts` (sibling to `copilotSystemPrompt` at `:15-54`):
  WhatsApp-appropriate tone, explains that sensitive actions require a PIN from
  the dashboard, no `[HUMANO]` handoff.
- Read tools (revenue, agenda, no-shows, free slots) execute directly and their
  results are sent back over WhatsApp.

### D.4 Sensitive actions — PIN step-up

`runAssistant` returns a `pendingAction` for sensitive tools (orchestrator
`:196-230`). The admin flow then:

1. Store `pendingActionId = pendingAction.id` and
   `pendingActionExpiresAt = now + 5 min` on the admin conversation. Reply:
   *"Para confirmar «{summary}», gere um PIN no painel (Configurações → WhatsApp
   Admin) e envie aqui. Ou responda 'cancelar'."*
2. On the next inbound message from an admin phone that has a live
   `pendingActionId`:
   - `"cancelar"` (case-insensitive) → `confirmSensitiveAction({ ..., reject: true })`,
     clear `pendingActionId`, reply confirmation.
   - a 6-digit PIN → verify with `rl:wa:adminpin:${shop.id}:${fromPhone}` (max 5
     / 10 min); check `sha256(pin) === adminPinHash` **constant-time** and
     `adminPinExpiresAt > now` and `pendingActionExpiresAt > now`. On success:
     `confirmSensitiveAction({ actionId: pendingActionId, barbershop, userId: ownerUserId })`,
     then **consume the PIN** (`adminPinHash = null`, `adminPinExpiresAt = null`),
     clear `pendingActionId`, reply the tool result. On failure: reply an error,
     keep the pending action, decrement attempts.
   - anything else while pending → re-prompt for the PIN (don't run a new AI turn
     that could create a second pending action).

The PIN is single-use and authorizes exactly the one pending action bound to
the conversation. Expiry on both the PIN and the pending action prevents stale
confirmations.

## Security analysis

- **Identity is server-verified.** The admin sender JID comes from
  Evolution/WhatsApp, not from message content, and `adminPhones` are writable
  only through the authenticated dashboard (`requireOwner`). A customer cannot
  self-elevate by typing.
- **Sensitive actions need physical dashboard access.** The PIN is generated in
  the authenticated web panel, short-lived (5 min), single-use, hashed at rest,
  attempt-rate-limited, and bound to one pending action — mitigating SIM-swap on
  the owner's number for state-changing operations.
- **Same tool boundary as everything else.** Admin tools are the copilot
  registry; execution goes through `confirmSensitiveAction` with server-built
  `ctx` and the **stored** input (model cannot re-supply args at confirm time).
- **Full audit.** Every admin turn and confirmation writes `AiActionLog` /
  `AuditLog`.
- **Residual risk (accepted, documented).** Read-only admin queries (e.g.
  revenue) are reachable with the registered number alone; a SIM-swapped number
  could read data (not change state). Hardening path for later: require an
  initial PIN to open a short admin "session" before any read tool. Not in v1.2
  per the chosen design (number = access, PIN = actions).
- **STT abuse bound.** 120s cap + existing WhatsApp rate limit cap transcription
  cost per phone; failures degrade to the text-only reply.

## Testing strategy

- **A (schema/config):** unit tests for `saveAdminChannelConfig` (phone
  normalization, instance-number rejection, dedupe) and `generateAdminPin`
  (hash stored, plaintext returned once, expiry set). Migration applies cleanly.
- **B (STT):** unit test `parseMessagesUpsert` audio-kind detection (mock
  `audioMessage` payloads) and the >120s guard; `transcribeAudio` with a mocked
  OpenAI client (success + error → fallback). `getBase64FromMedia` with a mocked
  `apiFetch`.
- **C (notifications):** integration test that `createAppointment` writes exactly
  one `Notification` per booking across all sources
  (`describe.skipIf(!DATABASE_URL)`); unit tests for `queries.ts` (unread count,
  tenant-fenced mark-read); `pushNewAppointmentToOwner` is a no-op when config is
  off (mocked evolution client asserts not-called).
- **D (admin channel):** unit tests for the admin routing predicate (phone ∈
  adminPhones), the PIN verification (correct/expired/wrong/attempt-limit,
  constant-time compare), and `confirmSensitiveAction` (atomic claim single-winner
  under double-confirm, reject path, stored-input execution). The web confirm
  route keeps its existing tests green after the extraction.
- Keep the suite green: `npm test`, `npm run typecheck`, `npm run lint`,
  `npm run build`.

## Out of scope (v1.2)

- Customer-facing payments / Stripe Connect (tracked separately in ROADMAP v1.1).
- SSE/live push for notifications (polling now; boundary preserved for later).
- TOTP-style always-on rotating code (chosen: explicit on-demand PIN).
- Admin "session" PIN for read tools (documented hardening path).
- Non-audio media understanding (image/OCR).

## New environment variables

- `OPENAI_TRANSCRIBE_MODEL` (default `gpt-4o-mini-transcribe`) — add to
  `.env.example` with a comment.

## Task grouping (for the plan)

- **Group A — Foundation:** schema + migration, `Channel` type, config server
  actions, settings UI card + PIN generator.
- **Group C — Notifications:** emit in `createAppointment`, queries, polling API,
  `NotificationBell`, best-effort WhatsApp push.
- **Group D — Admin channel:** `confirm-action.ts` extraction (+ keep web route
  green), admin routing/debounce/rate-limit, admin prompt, PIN step-up state
  machine.
- **Group B — STT:** Evolution media probe + `getBase64FromMedia`, parser
  `kind`, `transcribe.ts`, pipeline audio branch.

Groups are independent; A lands first because C and D use its fields.
